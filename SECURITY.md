# Security Specification: Stephen Hawking Brain

This document outlines the security architecture and data integrity protocols for the Stephen Hawking Brain research platform.

## 1. Security Overview
The platform utilizes a Zero-Trust Architecture built on top of Google Cloud and Firebase. Security is enforced at the database level using Attribute-Based Access Control (ABAC), ensuring that client-side code can never bypass protocol-defined invariants.

## 2. Authentication & Identity
- Provider: Google OAuth 2.0.
- Verification: Write operations to the "Hawking Ledger" (Global Pulse) strictly require `email_verified == true`.
- Identity Integrity: The user's `uid` is cross-referenced in every write operation. The Firestore Security Rules enforce that `request.auth.uid` must match the `userId` field in the incoming payload.

## 3. Data Integrity & Validation
All data saved to the archival systems must pass a Strict Schema Blueprint.

### Invariants for `logs` (Swarm Pulse):
1. Immutable Records: Logs are "Write-Once." Updates and modifications to existing logs are forbidden to maintain a scientific audit trail.
2. Schema Enforcement: Every log entry must contain exactly the required keys (`query`, `consensus`, `timestamp`, `userId`, `confidence`).
3. Size Constraints:
   - query: max 1,000 characters.
   - consensus: max 10,000 characters.
4. Temporal Integrity: Timestamps must be generated by the server (`request.time`). Client-provided timestamps are rejected.

## 4. The "Eight Pillars" of Hardened Rules
Our Firestore Security Rules are built on these core principles:
1. The Master Gate: Relational sync between users and their data.
2. Anti-Update-Gap: Validation helpers are called on every creation/write.
3. Identity Spoofing Guard: Path variable hardening for document IDs.
4. Denial of Wallet Protection: Evaluation order prioritized to minimize recursive cost-attacks (Auth -> Static Validation -> Document Reads).
5. PII Isolation: Sensitive user data is isolated from public research logs.
6. Implicit vs Explicit Check: No reliance on undefined fields; exact key matching.
7. Terminal State Locking: Archived research cannot be retracted or modified once consensus is reached.
8. Secure List Queries: Listing operations are restricted to ensure users only scan permissible indices.
